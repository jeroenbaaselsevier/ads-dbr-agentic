# Runbook: Agent Maintenance

This is the primary reference document for the **agent-trainer** role. Read it
before performing any maintenance or knowledge-promotion task.

---

## Repo architecture

### Canonical vs generated layers

```
agent-core/          ← CANONICAL. Hand-edited source of truth.
platform/            ← Templates (Jinja2-style). Hand-edited.
                        One sub-dir per platform: copilot/, claude/, codex/

.github/agents/      ← GENERATED (Copilot). Never hand-edit.
.claude/             ← GENERATED (Claude). Never hand-edit.
.agents/             ← GENERATED (Codex skills). Never hand-edit.
.codex/              ← GENERATED (Codex agents). Never hand-edit.
AGENTS.md            ← GENERATED (Codex root). Never hand-edit.
CLAUDE.md            ← GENERATED (Claude root). Never hand-edit.

scripts/             ← Build, validate, lifecycle, and promotion scripts.
tools/               ← Stable CLI tool layer (Databricks, AWS, local, QA).

agent-improvement/
  inbox/             ← Raw intake: captured lessons land here.
  triage/            ← Intermediate triage artifacts.
  reports/           ← Triage and promotion reports.
  schemas/           ← Lesson and promotion YAML schemas.
  promotions/        ← Reviewed promotion records (audit trail).

projects/<id>/       ← Project-scoped work, context, lessons, decisions.
```

### Key invariant

**Only `agent-core/` and `platform/` are source-of-truth.** Everything under
`.github/`, `.claude/`, `.agents/`, `.codex/`, `AGENTS.md`, and `CLAUDE.md` is
generated by `scripts/build_agent_adapters.py` and carries an `AUTO-GENERATED`
banner.

## File-target routing matrix

When promoting a lesson, choose the **closest valid layer**:

| Lesson type | Target file(s) |
|---|---|
| Project-only decision | `projects/<id>/context/decisions.md` or `context/lessons/` |
| Client-specific preference | `agent-core/profiles/clients/<client>.md` |
| User-specific preference | `agent-core/profiles/users/<user>.md` |
| Dataset / schema fact | `agent-core/catalog/tables/<table>.yaml` + `agent-core/references/<table>.md` |
| Join / filter / query caveat | Table manifest and/or relevant recipe |
| Reusable analysis workflow | `agent-core/recipes/<analysis>.md` |
| Cross-cutting operational rule | `agent-core/rules/core-rules.md` |
| Auth / environment rule | `agent-core/rules/security-and-auth.md` or relevant runbook |
| Output packaging convention | `agent-core/rules/output-contract.md` or `results-packager` role |
| Tool usage contract | `agent-core/tool-contract/*.yaml` |
| Architecture / maintenance procedure | `agent-core/runbooks/agent-maintenance.md` (this file) |
| New recurring failure mode | Eval task under `agent-core/evals/tasks/` |

### Principle

Global rules (`core-rules.md`) should remain **small and high-signal**. Do not
dump every fact there. Put detailed knowledge in the file closest to where it
applies.

## Update sequence

After any canonical change, run these steps in order:

### 1. Validate canonical consistency

```bash
python scripts/validate_agent_core.py
```

Checks: reference paths exist, table manifests present, roles exist, tool
contracts exist, runbooks exist, improvement structure intact.

### 2. Regenerate platform adapters

```bash
python scripts/build_agent_adapters.py
```

Reads `agent-core/` and `platform/` templates, writes all generated platform
files.

### 3. Validate generated outputs

```bash
python scripts/validate_platform_outputs.py
```

Checks: all expected generated files exist and carry the `AUTO-GENERATED`
banner.

### 4. Run evals (if behaviour changed)

If the promotion changes how the agent reasons, runs queries, or interprets
data:

- Check for existing eval tasks under `agent-core/evals/tasks/`.
- Run relevant evals to catch regressions.
- If no eval covers the change, create one.

## Promotion criteria

A lesson may be promoted when **at least one** of these is true:

1. The user explicitly requests promotion now.
2. The same issue has occurred across multiple projects.
3. The lesson has **high confidence** and **medium or high impact**.
4. The lesson closes a known architecture gap (e.g., missing manifest).
5. The lesson adds a new supported dataset or analysis type.

If none apply, the lesson remains in the inbox or project context for future
review.

## Promotion workflow

### Draft mode (default)

```bash
python scripts/promote_lessons.py \
    --from-inbox agent-improvement/inbox/2026-03-18_foo.yaml \
    --output agent-improvement/promotions/PRO-2026-03-18-001.yaml
```

Produces a promotion record with proposed target files and actions. Does not
modify canonical files.

### Apply mode (safe categories only)

```bash
python scripts/promote_lessons.py \
    --from-inbox agent-improvement/inbox/2026-03-18_foo.yaml \
    --output agent-improvement/promotions/PRO-2026-03-18-001.yaml \
    --apply
```

Auto-applies changes to safe categories:

- Profile file updates
- Reference doc updates
- Runbook updates
- Recipe doc updates
- Table manifest additions

**Does not** auto-apply edits to `core-rules.md`. Those require human review.

## Catalog completeness

Every entry in `agent-core/catalog/knowledge-index.yaml` must have:

1. A `knowledge-index.yaml` entry with `name`, `reference`, `spark_table`,
   `primary_key`, `keywords`, and `summary`.
2. A table manifest at `agent-core/catalog/tables/<name>.yaml`.
3. A reference doc at `agent-core/references/<name>.md`.
4. Optional: recipe additions and eval tasks.

`scripts/validate_agent_core.py` will fail if any of these are missing.

---

## Memory layers

The repo has four progressively broader memory layers:

| Layer | Scope | Location | Persistence |
|---|---|---|---|
| Project context | Single project | `projects/<id>/context/` | Project lifetime |
| Client profile | One client/account | `agent-core/profiles/clients/` | Indefinite |
| User profile | One analyst/user | `agent-core/profiles/users/` | Indefinite |
| Global rules | All projects | `agent-core/rules/`, `catalog/`, `references/`, `recipes/` | Indefinite |

Always choose the **narrowest** layer that covers the lesson.

## Examples

### Example 1: Project-only fact

> "For this project, the client wants results grouped by NUTS-2 region."

→ Write to `projects/<id>/context/decisions.md`. Do not promote.

### Example 2: Client preference

> "Client X always wants country-first framing in summary charts."

→ Write to `agent-core/profiles/clients/x.md`.

### Example 3: Schema caveat

> "Rosetta publisher fields should use the current unsuffixed columns."

→ Update `agent-core/references/rosetta.md` and
`agent-core/catalog/tables/rosetta.yaml`.

### Example 4: Reusable analysis pattern

> "The journal trend analysis pattern works well — make it a recipe."

→ Create `agent-core/recipes/journal-trend.md` with YAML frontmatter.

### Example 5: Global operational rule

> "Always warn users when coverage is below 50%."

→ Update `agent-core/rules/core-rules.md` (after eval review).
